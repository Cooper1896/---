# 页面开发手册：Zenless Tavern

## 适用范围

本手册用于规范后续所有页面的新增、重构与接入方式，保证整个项目在布局、视觉、架构和维护方式上保持一致。

## 新页面必须包含的内容

每个页面在落地前都要明确这几项：

- 页面名称与模块 ID
- 页面职责
- 属于桌面工作区、手机 iframe 内页，还是双端联动页
- `Model`
- `Update`
- `View`
- Zod schema
- 使用到的共享模板
- 需要接入到底部系统面板的状态或入口

## 新页面标准流程

1. 先写页面目标，只保留一个主要任务。
2. 先判断页面属于哪一层：桌面层、手机层、系统层。
3. 先接入共享布局骨架，不允许裸写独立页面。
4. 先补 Zod schema，再接收页面变量、配置或表单输入。
5. 先拆出 `Model / Update / View`，再写具体内容。
6. 先复用共享 EJS 模板和系统组件，再决定是否新增局部组件。
7. 接入底部系统面板的状态映射或快捷操作。
8. 最后做视觉微调和动效，不允许一上来只画皮肤。

## 视觉一致性检查

新增页面时必须逐项检查：

- 是否仍然一眼看出属于同一个产品
- 是否沿用了相同的边框、分隔、字号和标签系统
- 是否沿用了相同的手机壳与 iframe 规范
- 是否保留了底部系统面板
- 是否继续使用同一套颜色语义
- 是否避免了“单页自创风格”

## 组件使用顺序

优先级固定如下：

1. 共享壳层组件
2. 共享模板
3. 共享主题 token
4. 共享基础组件
5. 页面局部组件

只有前四层无法满足时，才新增页面私有组件。

## 禁止事项

- 禁止页面直接硬编码一套新颜色。
- 禁止页面私自更换主要布局结构。
- 禁止把底部系统面板删掉或弱化成无关元素。
- 禁止把手机 iframe 做成装饰图。
- 禁止把状态、事件、模板、样式全部塞进一个文件。
- 禁止复制已有页面后只改文案，不做模块抽象。

## 页面完成标准

一个新页面只有在下面这些条件满足后，才算完成：

- 功能职责清晰
- 接入共享布局成功
- 接入统一主题 token
- 页面输入有 Zod 校验边界
- 页面结构符合 MVU 拆分
- 手机 iframe 与桌面层关系明确
- 底部系统面板已联动
- 命名、目录和模块边界可维护

## 重构优先级

如果后续需要整理当前仓库，建议顺序如下：

1. 先统一主题 token
2. 再统一外层布局骨架
3. 再抽离 iframe 手机壳
4. 再抽离底部系统面板
5. 再把页面逐步迁移到 MVU + Zod + EJS 组织方式

这样做风险更低，也更便于逐步验证页面风格是否稳定。
