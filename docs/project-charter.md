# 项目章程：Zenless Tavern Frontend

## 目标

本项目的目标不是做一个普通的聊天页，而是做一个基于《绝区零》世界观的类酒馆前端体验。整体应参考 SillyTavern 的交互深度与可扩展性，但在视觉、叙事和界面结构上形成统一的 `Zenless Tavern` 风格。

后续创建的所有页面，都必须继承同一套布局骨架、视觉语言、交互语气和模块组织方式，不能把主页做成一个方向、子页面又变成另一个方向。

## 不可变约束

- 世界观固定为《绝区零》。
- 产品方向固定为“类酒馆”前端，不做通用后台或普通站点。
- 页面技术基线固定为 `MVU + Zod 变量 + EJS 脚本 + iframe 手机 UI + 底部系统面板`。
- 大部分交互能力、模块划分思路、可扩展性设计，可以参考 [SillyTavern](https://github.com/SillyTavern/SillyTavern)。
- 后续任何新增页面，都必须先复用共享骨架，再写页面内容。
- 开发过程必须有序，可拆分、可复查、可维护，避免一次性堆砌。

## 产品体验定义

### 核心气质

- 不是泛赛博朋克，而是《绝区零》的“都市、工业、终端、信号、街区系统感”。
- 不是纯展示站，而是一个可操作、可扩展、带系统状态感的“终端界面”。
- 不是堆满卡片的仪表盘，而是有主次层级、有信息压强、有留白控制的工作台。

### 参考方向

- 顶层体验参考 SillyTavern 的工作区心智。
- 视觉表达参考《绝区零》的 UI 气质，而不是直接照搬游戏 HUD。
- 界面必须保持“系统终端 + 手机内页 + 底部控制层”的统一组合。

## 统一布局骨架

所有页面默认遵循同一套外层结构：

1. 桌面级主容器
2. 顶部标题/状态区
3. 中央主工作区
4. `iframe` 手机 UI 容器
5. 底部系统面板

### 布局规则

- `iframe` 手机 UI 是核心舞台，不是附属装饰。
- 底部系统面板必须全站统一，承担系统状态、快捷操作、上下文切换或日志反馈。
- 顶部、主区、底部之间的间距、边线、层级关系必须稳定，不允许每页重新定义。
- 新页面只允许在共享骨架内替换主内容，不允许私自改外层结构。

## 视觉系统基线

### 视觉主题

- 主体氛围：新艾利都、终端、工业、讯号、都市运营界面。
- 风格关键词：克制、锐利、带设备感、带广播感、带系统编排感。
- 禁止做成泛用霓虹模板、通用后台模板、AIGC 常见赛博风拼贴。

### 颜色策略

- 必须有一套全局语义色，而不是页面各自挑颜色。
- 默认保留 1 个主强调色、1 个辅助强调色、1 套中性色阶、1 套状态色。
- 所有颜色应通过变量管理，不能在页面里零散硬编码。

建议语义层：

- `bg`: 主背景
- `surface`: 面板背景
- `line`: 分隔线/边框
- `text`: 主文本
- `muted`: 次级文本
- `accent-primary`: 主强调
- `accent-secondary`: 次强调
- `danger`, `warning`, `success`, `info`: 状态反馈

### 字体与排版

- 全站最多两套字体。
- 标题、编号、系统标签必须有明显的终端感与识别度。
- 正文与表单文本优先保证可读性，不能只追求造型。
- 标题层级、标签大小、系统码样式必须统一，不允许页面自创一套。

### 动效

- 动效应该强化“系统存在感”和层级切换感。
- 允许使用入场、扫描、状态闪烁、轻量 hover、面板滑入等动效。
- 动效要克制，不能影响阅读、录屏体验和手机 iframe 的稳定性。

## 架构约束

### MVU

页面功能按 `Model / View / Update` 组织：

- `Model`: 页面状态、派生状态、数据快照
- `View`: 纯视图拼装与模板渲染
- `Update`: 事件、动作、状态迁移

禁止把大量状态变更、网络请求和视图判断混写在一个页面文件里。

### Zod 变量

- 所有主题变量、页面配置、表单输入、外部数据入口，优先通过 Zod schema 定义。
- 变量要有默认值、校验规则、可追踪的类型边界。
- 新页面不允许直接依赖“隐式对象结构”。

### EJS

- EJS 作为模板脚本层，用于稳定输出结构片段和可复用页面模板。
- 页面脚本组织要模块化，避免在单文件中混杂模板、数据、事件和样式约定。
- 共享布局、系统面板、手机壳、模块头部等，应优先沉淀为可复用模板。

### iframe 手机 UI

- 手机 UI 作为独立体验层处理，避免桌面工作台和手机内页逻辑互相污染。
- 手机 iframe 的尺寸、边框、比例、留白、遮罩、标题栏规则必须全站统一。
- 所有手机内页都要继承同一套主题变量和组件约束。

### 底部系统面板

- 底部系统面板是产品级固定模块，不是某一页私有元素。
- 它应承担全局状态、快捷入口、运行信息、切换入口或提示反馈。
- 面板内容可以扩展，但框架、节奏、视觉权重必须一致。

## 可维护性原则

### 目录组织

后续实现应逐步向以下结构靠拢：

- `src/core`: 全局框架、应用骨架、路由、共享上下文
- `src/features`: 页面或功能模块
- `src/schemas`: Zod schema 与变量定义
- `src/templates`: EJS 模板与共享片段
- `src/ui`: 共享 UI 组件、壳层、底部面板、手机框架
- `src/theme`: 设计 token、颜色、字体、间距、动效变量
- `src/mvu`: 通用 MVU 辅助工具

### 复用策略

- 能做成共享 token，就不要写死在页面。
- 能做成共享模板，就不要复制页面结构。
- 能做成共享壳层，就不要每页自己拼布局。
- 能用 schema 固定边界，就不要靠约定俗成。

### 页面开发纪律

- 先定义页面职责，再写结构。
- 先接入共享骨架，再填充页面内容。
- 先定义 schema，再接表单和数据。
- 先复用系统组件，再考虑新增组件。
- 每个页面必须能独立维护，也必须能无缝并入全站。

## 页面一致性要求

以后任何新页面都必须满足以下条件：

- 与主页共享同一视觉语法。
- 与主页共享同一布局骨架。
- 与主页共享同一系统面板。
- 与主页共享同一手机 iframe 容器规则。
- 与主页共享同一 token、字号、边框、间距与动效原则。

如果某个页面看起来像是另一个项目的一部分，那就是不合格。

## 当前仓库的定位

当前仓库更接近一个 React/Vite 原型。它可以继续作为探索基础，但从现在开始，后续迭代应以本章程为约束，逐步收敛到目标技术路线与统一设计系统，而不是继续无序生长。

## 后续执行方式

后面每次新增页面或重构模块时，都以本章程作为默认准则。若要偏离，必须先说明原因，再决定是否调整全局规则，而不是只在某个页面里临时例外。
